# Finance Tracker

Personal finance web app built with TanStack Start, BetterAuth, Drizzle ORM v1, and Turso (libSQL/SQLite edge).

## Project Status

Early stage. The project has been bootstrapped with TanStack Start, TanStack Query, Tailwind CSS v4, and Biome. Only the root route (`/`) exists with a placeholder component. All features described in this file are to be built according to the PRD (`prd.md` at project root).

## Data Fetching Rules

These patterns are **non-negotiable**. Every data operation must follow them exactly.

### 1. Query/Mutation definitions live in `src/queries/`

Every query and mutation is defined as a standalone options object using `queryOptions()` or `mutationOptions()` from `@tanstack/react-query`. Group by domain:

```
src/queries/
├── expenses.ts      # expense query/mutation options
├── earnings.ts      # earning query/mutation options
├── categories.ts    # category query/mutation options
├── dashboard.ts     # dashboard-specific aggregation queries
└── auth.ts          # session/user query options
```

Example pattern:

```typescript
// src/queries/expenses.ts
import { queryOptions, mutationOptions } from "@tanstack/react-query";
import { getExpenses, createExpense } from "#/server/expenses";

export const expensesQueryOptions = (filters?: ExpenseFilters) =>
  queryOptions({
    queryKey: ["expenses", filters],
    queryFn: () => getExpenses(filters),
  });

export const createExpenseMutationOptions = () =>
  mutationOptions({
    mutationFn: createExpense,
    onSuccess: (_data, _variables, _context, queryClient) => {
      queryClient.invalidateQueries({ queryKey: ["expenses"] });
      queryClient.invalidateQueries({ queryKey: ["dashboard"] });
    },
  });
```

### 2. Route loaders only warm the cache

Loaders call `queryClient.ensureQueryData()`, `prefetchQuery()`, or `fetchQuery()`. They must **never return data directly**. This keeps TanStack Query's cache as the single source of truth.

```typescript
// In a route file
export const Route = createFileRoute("/_app/expenses")({
  loader: ({ context }) => {
    context.queryClient.ensureQueryData(expensesQueryOptions());
  },
  component: ExpensesPage,
});
```

### 3. Components consume data via hooks

Use `useQuery()` or `useSuspenseQuery()` in components. Never call `fetch()` or server functions directly from a component.

### 4. Mutations use `useMutation()` with cache invalidation

All writes go through `useMutation()`. The `onSuccess` callback must invalidate every query key that could be affected by the mutation. Think through which queries need invalidation — e.g., creating an expense should invalidate both the expenses list and dashboard aggregations.

### 5. Server functions use `createServerFn`

All server-side logic (database queries, auth checks) is defined using TanStack Start's `createServerFn`. These are the functions that `queryFn` and `mutationFn` call. They run on the server only.

## Database (Drizzle ORM + Turso)

### Schema overview

All tables are defined in `src/db/schema.ts` using Drizzle's SQLite column types from `drizzle-orm/sqlite-core`. Use UUIDs (text) for primary keys, integer timestamps for dates (`mode: "timestamp"`).

| Table          | Purpose                                                       |
| -------------- | ------------------------------------------------------------- |
| `user`         | Managed by BetterAuth. Extended with `currency` text column.  |
| `session`      | Managed by BetterAuth.                                        |
| `account`      | Managed by BetterAuth.                                        |
| `verification` | Managed by BetterAuth.                                        |
| `categories`   | User-scoped expense labels. Has optional `budget` (real).     |
| `expenses`     | Money out. FK to `categories` and `user`. Has recurrence.     |
| `earnings`     | Money in. FK to `user`. Has recurrence. No category.          |

### Schema rules

- All custom tables (categories, expenses, earnings) have `userId` as a NOT NULL FK to `user.id`.
- `createdAt` and `updatedAt` use integer columns with `mode: "timestamp"`. `updatedAt` uses `.$onUpdate()`.
- `recurrence` is a text column constrained to: `"none"`, `"weekly"`, `"monthly"`, `"yearly"`. Default: `"none"`.
- `recurrenceEndDate` is a nullable integer timestamp.
- `amount` is `real`, always positive. Enforce positivity at the application layer, not the DB.
- `isDefault` and `isActive` on categories are integer columns used as booleans (0/1). SQLite has no native boolean.
- Never write raw SQL. Always use Drizzle's query builder or relational queries.
- Never edit files in `src/db/migrations/` by hand. Run `pnpm db:generate` to create migrations from schema changes.

### Default categories

Seed these for every new user on signup: **Food, Transport, Entertainment, Bills & Utilities, Shopping, Health, Education, Other**. Mark them with `isDefault: true`. Default categories can be hidden (`isActive: false`) but never deleted.

## Authentication (BetterAuth)

- **Email + password only.** No OAuth, no magic links, no email verification in v1.
- BetterAuth manages `user`, `session`, `account`, and `verification` tables.
- The `_app/route.tsx` layout checks the session. If expired or missing, redirect to `/login`.
- On signup: auto-login, seed default categories, redirect to `/app`.
- On login: redirect to `/app`.
- On logout: clear session, redirect to `/`.
- Never expose raw session tokens in client-side code.

## Recurrence Logic

Recurring transactions are **generated on read** (virtual expansion), not pre-inserted in the database. The DB stores only the base transaction template with its recurrence rule.

### Expansion rules

- Generate occurrences from the transaction's `date` up to `min(today, recurrenceEndDate)`.
- Step by the recurrence interval: 7 days (weekly), 1 month (monthly), 1 year (yearly).
- Users edit/delete the **template** (base transaction). Individual occurrences cannot be modified independently in v1.
- Dashboard aggregations and transaction lists must account for expanded recurrences when computing totals and displaying entries.
- Each generated occurrence should carry a reference to its parent template ID and its computed date, so the UI can distinguish templates from occurrences.

## Currency

- Display-only formatting. **No currency conversion, ever.**
- Store raw numbers in the DB. Format with the user's `currency` setting at render time.
- The `currency` column on the user table stores an ISO 4217 code. Default: `"USD"`.
- Supported currencies for v1: USD, BRL, EUR, GBP, CAD, AUD, JPY.
- Use `Intl.NumberFormat` for locale-aware currency formatting.
- Prompt new users to select a currency during onboarding, or default to USD with the option to change in settings.

## Code Style

### TypeScript

- Strict mode is enabled (`strict: true` in tsconfig). Never use `any` — type everything explicitly.
- Use `interface` for object shapes. Use `type` only for unions, intersections, or mapped types.
- Named exports only. No `export default` except for route files (required by TanStack Router's file-based routing).

### Formatting (Biome)

- **Indentation: tabs** (not spaces).
- **Quotes: double quotes** for JS/TS strings.
- **Import organization: automatic** (Biome `organizeImports` is enabled).
- Run `pnpm check` to lint and format in one pass. Fix all issues before considering a task done.

### Styling

- **Tailwind CSS v4 only.** No inline `style={}`, no CSS modules, no external CSS files (except `globals.css` for Tailwind directives).
- Use shadcn/ui components for all UI elements. Do not build custom UI primitives when a shadcn equivalent exists.
- Framer Motion for animations: page transitions, modal/sheet entrances, micro-interactions (hover effects, button press).
- Design aesthetic: minimal, whitespace-heavy, muted color palette. Inspired by Linear, Vercel, Raycast.

### Component conventions

- Co-locate small, route-specific components in the route file. Extract to `src/components/` only when reused across multiple routes.
- shadcn/ui primitives live in `src/components/ui/` (installed via shadcn CLI).
- Shared, composed components (e.g., `ExpenseForm`, `TransactionTable`) live in `src/components/`.

## Boundaries

### Never do these

- Return data directly from TanStack Router loaders (loaders only warm the query cache)
- Write raw SQL — always use Drizzle query builder
- Edit files in `src/db/migrations/` by hand
- Edit `src/routeTree.gen.ts` — it is auto-generated by TanStack Router
- Build custom UI components when a shadcn/ui equivalent exists
- Implement currency conversion (display formatting only)
- Add OAuth, email verification, or password reset (out of scope for v1)
- Pre-insert recurring transaction occurrences into the DB (always expand on read)
- Use `npm`, `bun`, or `yarn` — this project uses `pnpm`
- Use `eslint` or `prettier` — this project uses Biome
- Use inline styles or CSS modules — use Tailwind classes only
- Use `export default` except in route files
- Skip cache invalidation after mutations

### Ask the user before doing these

- Adding a new npm dependency not already in `package.json`
- Adding a new shadcn/ui component not already in `src/components/ui/`
- Schema changes that require a new migration
- Creating or modifying environment variables
- Changing BetterAuth configuration or internals
- Modifying `biome.json`, `tsconfig.json`, or `vite.config.ts`
- Any architectural decision not covered by this document or the PRD

### Always do these

- Run `pnpm check` before considering a task done (catches lint + format issues)
- Invalidate the correct React Query keys after every mutation (think through all affected queries)
- Define all queries and mutations via `queryOptions()` / `mutationOptions()` factories in `src/queries/`
- Use `createServerFn` for all server-side logic
- Follow the route loader → query cache → component hook data flow
- Seed default categories for new users on signup
- Handle loading states (skeletons/spinners) and error states (toast notifications) for async operations
- Ensure all forms have both client-side validation and server-side validation
- Use proper ARIA labels and keyboard navigation for accessibility
